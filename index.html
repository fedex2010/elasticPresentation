Que es Elasticsearch ???

  • Es un motor de busquedas full text y analizador de datos, open source y altamente escalable

  • Está construido sobre Apache Lucene ( fue el primer motor de busqueda full text mas popular del mercado)

  • Desarrollado en Java

  • La primera version salio en febrero del 2010

  • En el fondo es un indice invertido

Porque es facil de escalar

    Elasticsearch está construido para escalar facilmente.
    En el momento que se necesite mas capacidad simplemente hay que añadir un nodo más, lo demas se acomoda solo

RESTful API

    Elasticsearch es API driven. Casi cualquier accion puede llevarse a cabo mediante una API RESTfull.

    Las respuesta siempre son en formato JSON, pueden ser leidas por cualquier lenguajes de programacion y por cualquier persona

Ultra manejable lenguaje de consultas

    Domain Specified Language (DSL)

    Su API REST ofrece un complejo pero versatil lenguaje para hacer consultas y facetados, es muy intuitivo y facil de usar.

    Cada consulta es simplemente un objecto JSON y puede contener cualquier tipo de consulta o varias de ellas

Arquitectura

Cluster:

Un cluster es un conjunto de uno o más nodos que mantienen toda la información de manera distribuida e indexada.
Cada cluster está identificado por un nombre, por defecto se llaman “elasticsearch”

Arquitectura

Nodo:

Un nodo es una instancia de Elastic que forma parte de un cluster, almacena tu información y ayuda con las tareas de indexación y búsqueda del cluster.
Los nodos están identificados por un nombre también.
Puede haber tantos nodos como quieras por cada Cluster, en caso de que no haya ningún Cluster configurado en el momento de creación este lo creará y se unirá a él.

Arquitectura

Indice:

Un Index es una colección de documentos que tienen características similares pero el indice en si no almacena los documentos, Cada índice está compuesto por uno o más shards.

Arquitectura

Sharding y Replicas:

Cada shard es una instancia de un índice de Lucene, que podría imaginarse como un motor de búsqueda en sí mismo, que indexa y administra consultas para un subconjunto de datos en un cluster de Elasticsearch.

Cada replica es un copia exacta de un indice y los shards replicas no pueden habitar en el mismo nodo de su shard originador

Analysis

  • Analysis es el proceso de conversión de texto, como el cuerpo de cualquier correo electrónico, en tokens o términos que se agregan a el índice invertido para la búsqueda.
  • Es un proceso que puede hacerse en tiempo de indexacion o de busqueda, para todos o algunos campos del documento
  • Elastic tiene un amplio repertorio de analizers que se adaptan a distintos contextos o lenguajes
  • Tambien se puede definir analizers custom o hacer una mezpla entre ellos

Anatomia de un analizer

Anatomia de un analizer

  • Character filter:

    Un Character filter recibe el texto original como una secuencia de caracteres y puede transformar la secuencia agregando, eliminando o cambiando caracteres. Por ejemplo, un filtro de caracteres se podría usar para convertir números árabe-hindúes (٠ ١٢٣٤٥٦٧٨ ٩) en sus equivalentes árabe-latinos (0123456789), o para eliminar elementos HTML como de la secuencia..

  • No es necesario

Anatomia de un analizer

  • Tokenizer:

    Un tokenizer recibe un flujo de caracteres, lo divide en tokens individuales, generalmente palabras individuales

  • debe haber al menos un tokenizer

Anatomia de un analizer

  • Token filter:

    Acepta los tokens devueltos por el tokenizer y aplica determinada transformaciones sobre cada uno

    Por ej: pasar todo a minuscula, hallar la raiz de las palabras (Stemming) o remover las stopwords entre otras muchas opciones

  • No necesariamente se debe agregar un token filter

Mapping

El mapeo es el proceso de definir cómo un documento, y los campos que contiene, se almacenan e indexan

Por ejemplo:

  • que campos deberían ser tratado como full text searcheables (string)
  • que campos contienen numeros, fechas, booleans
  • definir tipos de datos especiales para tratar datos como geolocalizacion, direcciones ip, percolators
  • definir si es un campo debe ser indexado o no procesado

Mapping

  • reglas para mapping dinamico
  • definir analyzers especificos para determinados campos
  • el tipo de dato de un campo no puede ser modificado luego de ser creado, sino hay que reindexar
